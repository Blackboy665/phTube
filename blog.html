Scope of `var`, `let`, and `const`

When working with JavaScript, understanding the differences in scope between var, let, and const is crucial for writing effective and bug-free code. Each of these keywords has unique characteristics that influence how and where a variable is accessible within your code.

var

Scope

Function-scoped: Variables declared with var are limited to the function within which they are declared. If declared outside any function, they have global scope.

Hoisting: Variables declared with var are hoisted to the top of their containing function or the global scope. This means you can reference a var variable before its declaration, but it will be undefined until the line where it's actually defined is reached.

console.log(x); // undefined
        var x = 5;
        console.log(x); // 5

Use Cases

Legacy code: Many older JavaScript codes use var. When maintaining or interacting with such codebases, understanding var is important.

Compatibility: If you're working in environments where let and const are not supported, var might still be your go-to.

let

Scope

Block-scoped: Variables declared with let are limited to the block, statement, or expression where they are declared.

Hoisting: Like var, let declarations are hoisted to the top of their block, but unlike var, they are not initialized. Accessing them before their declaration results in a ReferenceError.

if (true) {
        let y = 10;
        console.log(y); // 10
        }
        console.log(y); // ReferenceError: y is not defined

Use Cases

Block-level scope needs: When you need a variable limited to a specific block, use let.

Loop counters: let is particularly useful in loops where each iteration needs its own scope.

for (let i = 0; i < 5; i++) {
        console.log(i);
        }
        console.log(i); // ReferenceError: i is not defined

const

Scope

Block-scoped: Like let, variables declared with const are limited to the block where they are defined.

Hoisting: const variables are also hoisted, but not initialized, similar to let.

Use Cases

Constant values: Use const when you need a variable whose value should not be reassigned.

Immutable references: While the value a const variable points to cannot be changed, if it points to an object or array, the contents of that object or array can be modified.

const z = 20;
        z = 25; // TypeError: Assignment to constant variable.
        
        const arr = [1, 2, 3];
        arr.push(4); // This is allowed
        console.log(arr); // [1, 2, 3, 4]

Use Cases of null and undefined

null

Intentional absence: null is an assignment value that represents no value or no object. It's explicitly set by the programmer to indicate "no value."

let foo = null;
        console.log(foo); // null

Use case: When you want to clear a variable or reset its value.

let obj = {name: 'Alice'};
        obj = null; // obj is now explicitly set to have no value

undefined

Default state: undefined means a variable has been declared but not yet assigned a value.

Automatic assignment: JavaScript assigns undefined to variables that are declared but not initialized.

let bar;
        console.log(bar); // undefined

Function arguments: If a function is called with fewer arguments than parameters, the missing parameters are undefined.

function greet(name) {
        console.log('Hello ' + name);
        }
        greet(); // Hello undefined

What is a REST API?

REST API (Representational State Transfer Application Programming Interface)

Architecture style: REST is an architectural style for designing networked applications. It relies on a stateless, client-server, cacheable communications protocol -- the HTTP.

Resource-based: In a RESTful application, resources (data) are identified by URIs (Uniform Resource Identifiers). Resources can be retrieved, created, modified, and deleted using standard HTTP methods (GET, POST, PUT, DELETE).

Characteristics

Stateless: Each request from a client to server must contain all the information the server needs to fulfill that request. The server does not store any state about the client session.

Cacheable: Responses must implicitly or explicitly define themselves as cacheable or not to prevent clients from reusing stale data.

Uniform interface: Simplifies and decouples the architecture, enabling each part to evolve independently. It is defined by four constraints: resource identification, resource manipulation through representations, self-descriptive messages, and hypermedia as the engine of application state (HATEOAS).

Example

A typical REST API endpoint might look like this:

GET /users/123

GET: HTTP method indicating the action (retrieve data).

/users/123: URI identifying the specific resource (user with ID 123).

Use Cases

Web services: REST APIs are commonly used in web services due to their simplicity and scalability.

Microservices: RESTful principles are a natural fit for building microservices architectures.
